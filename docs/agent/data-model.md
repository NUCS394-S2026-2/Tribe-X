Owned by **Tribe_X_arch_design**. Read before writing any code that touches Firestore.

## Shared `User` Type — define in Iteration 0

```typescript
// Move to src/shared/types/User.ts once finalized
export interface User {
  uid: string; // Firebase Auth UID — primary key
  email: string;
  displayName: string;
  // TODO: add fields as guild decides
}
/** 
 * /tracks/{trackId}
 * Represents the uploaded MP3 file and its processing status.
 */
export interface Track {
  trackId: string;       // Primary Key
  ownerId: string;       // Reference to User.uid
  fileName: string;      // Original name of the file
  storageUrl: string;    // URL to the MP3 in Firebase Storage
  status: 'pending' | 'analyzing' | 'completed' | 'error';
  uploadedAt: Date;
}

/** 
 * /tags/{trackId}
 * Stores the results generated by the AI Agent Interface.
 */
export interface MusicTags {
  trackId: string;            // Reference to Track.trackId
  genres: string[];           // e.g., ["Cinematic", "Electronic"]
  instruments: string[];      // e.g., ["Synthesizer", "Drums"]
  vocalTraits: string[];      // e.g., ["Male", "High-pitched", "Aggressive"]
  soundsLike?: string[];      // Stretch goal: Similar artists
  confidenceScore: number;    // Accuracy of the AI analysis
  lastUpdated: Date;
}
```

## Firestore Collections

Document each collection: path, document ID strategy, TypeScript type, subcollections.

```

## Firestore Collections

### /users/{uid}
- **Document ID**: Firebase Auth UID
- **Type**: `User`
- **Purpose**: Core user profile.

### /tracks/{trackId}
- **Document ID**: Auto-generated or file hash.
- **Type**: `Track`
- **Purpose**: Metadata and status for uploaded MP3 files.

### /tags/{trackId}
- **Document ID**: Matches the corresponding `trackId`.
- **Type**: `MusicTags`
- **Purpose**: Results from the AI analysis (genres, instruments, etc.).
```

Add a `###` section per collection; delete placeholders once real collections are defined.

## State Management Rules

1. **Firestore is source of truth.** Don't duplicate Firestore data in local state without caching intent.
2. **React Context** for session-scoped state (auth user, theme). Use Firestore listeners for domain data.
3. **Props** for component-local data. Avoid drilling beyond two levels — lift to context or Firestore.

## Serialization Rules

- Convert Firestore `Timestamp` to `Date`/ISO string before entering React state.
- Use `null` (not `undefined`) for absent optional fields in Firestore.
- Use a fixed `Date` in tests — never `serverTimestamp()`.
